Required reading: The 50% Rule is a principle of maximizing success by combining multiple methods or approaches, each with an individual success rate exceeding 50%, to increase the overall probability of achieving the desired outcome. Think of it like flipping a weighted coin that lands heads more than half the time: by flipping the coin repeatedly, you dramatically improve your odds of getting the result you want, even though no single flip guarantees success. When applied to language models, the 50% Rule encourages using multiple methods of communication, phrasing, and conceptual representation in a single task. Each method alone is likely to work in more than half the cases (e.g., simple tags, natural language descriptions, or stylistic expansions), but when combined and layered redundantly, the overall winrate increases significantly because the diverse representations help ensure the intent is understood.

How this applies to our codebase is in commenting, function naming, variable naming, and all the other token based cues that will direct the future models and people within the codebase to a "win" ie to rapid comprehension of the material. As such, we want to maximize the number of cues via careful comments that examine the material from angles that might not be explicitly outlined just by looking at the code alone, adding context and diversifying the tokens to allow for better recognition and generalization of the logic and programmatic patterns. In other words, explain your work by doing things like annotating all functions with explanations behind the "why" they exist in the context of the application, implicitly explaining the "what" that they do. Apply this also to inline comments within larger functions: multiple layers of cues allow future reviewers to better contextualize the information and understand the codebase.

Other guidance:
**Design for testability**. Every new behavior must be verifiable with automated tests.
**Regression tests are mandatory** for **all code changes** (feature, refactor, or bug fix).
**User-reported bugs are systemic**: they indicate (a) a code defect and (b) a *testing gap*. We fix both.
**Functional Programming**: Keep functions **pure and side‑effect free**; No side effects.


**Monolithic, not module driven**: Prefer a **single source file** for application logic (e.g., `src/script.js`). Organize by regions/sections inside the file rather than splitting utilities across many modules. In any situation where a single file is possible, it is preferable to multi-file constructions. In other words, keep **utilities co‑located** and avoid fragmentation across modules. 
That being said, splitting off long lists or other data, such as long text strings or other material, into their own data file is desireable, the goal there being to gather together all the code in one place so other model's can more easily review it and see patterns (while random, specific data interferes with that, and also doesn't fit well with the "functional programming" ideal, as essentially preset data implies a sort of constant: in such a case an imported data that we can then abstract into a well named variable, to further increase context cues per the 50% rule, is much more ideal).



Testing Doctrine:
Every code change must add or update regression tests.
A bug indicates, above all else, a failure of our testing and testing methodology. In other words, had our testing been more complete from the start, we would have caught this bug, bug X, and as such, our first failure in the code occurred with our test design. This implies that when we are faced with a bug, our first process should be to evaluate what we did wrong when creating our testing, and then not only add tests for these specific instances, but find the general class of failure in our testing method and expand it broadly. As such, a single bug fix will often lead to us idenfifying other as yet unidentified bugs by guiding us in where our existing net doesn't have full coverage.

In summation, our coding paradigm is test-first, and as such, we treat testing as a first class citizen, where the majority of our thoughts and efforts belong. To successfully produce excellent code, our number one priority is always to maximize our testing, and in this regard we will always pursue tests beyond the user's bare requests, creatively basing our tests on the intended input>output of the program itself and attempting to capture fully future bugs that may not have yet been identified.

Lastly, testing needs to target user surfaces and real world conditions. If there's is a gui, its not enough to confirm a given function used BY the gui is correct, you need to actually populate the GUI with input and press buttons on it, confirming that ultimately the behavior is correct. Testing needs to be multilayered.
